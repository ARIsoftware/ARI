Pull the latest ARI updates from the official upstream repository.

## Steps

### 1. Migration Check (one-time, for upgrades from pre-tracked launcher)

The launcher files (`./ari`, `./ari.cmd`, `.ari/cli.js`) used to be gitignored and generated by the installer. They are now tracked upstream. If a user is on an older install, those files exist on disk as **untracked** and will block `git pull` with an "untracked working tree files would be overwritten by merge" error.

Check whether each of the three files is tracked in git:

```bash
git ls-files --error-unmatch ari ari.cmd .ari/cli.js 2>/dev/null
```

If the command **exits non-zero** (one or more files are not tracked) AND the file exists on disk (`test -e ari`, `test -e ari.cmd`, `test -e .ari/cli.js`), the user is on an older install and the upcoming pull will fail.

Tell the user:
> "Detected untracked launcher files from a previous install. They're now tracked upstream and would block the merge. Safe to remove them and let git restore the latest versions?"

On confirmation, run:
```bash
rm -f ari ari.cmd .ari/cli.js
```

Skip silently if the files are already tracked, or don't exist.

### 2. Pre-flight Checks

- Verify we are inside a git repository. If not, stop and tell the user.
- Check if an `upstream` remote exists by running `git remote -v`.
  - If `upstream` does NOT exist, add it:
    ```
    git remote add upstream https://github.com/ARIsoftware/ARI.git
    ```
    Tell the user you've added the upstream remote.
  - If `upstream` exists but points to the wrong URL (not `ARIsoftware/ARI`), warn the user and ask if they want to update it.

### 3. Check for Uncommitted Changes

- Run `git status` to check for uncommitted changes (staged, unstaged, or untracked).
- If there are uncommitted changes, warn the user:
  > "You have uncommitted changes. It's recommended to commit or stash them before updating. Would you like to continue anyway?"
- Ask for confirmation. If they say no, stop.

### 4. Fetch Upstream

- Run `git fetch upstream` to download the latest changes.
- If the fetch fails (e.g., network error, repo not accessible), show the error and stop.

### 5. Show What's Changed

- Run `git log HEAD..upstream/main --oneline` to show new commits.
- Run `git diff --stat HEAD..upstream/main` to show which files changed.
- If there are no new changes, tell the user "You're already up to date!" and stop.
- Present the summary to the user clearly:
  > "There are X new commits from ARI upstream affecting Y files."
  > Show the commit list and file summary.

### 6. Ask for Confirmation

Ask the user:
> "Would you like to merge these updates into your local branch?"

With options:
1. **Yes, merge updates** — Proceed with the merge
2. **No, cancel** — Stop without making changes

If they say no, stop.

### 7. Merge Upstream

- Run `git merge upstream/main --no-edit` to merge the updates.
- If the merge succeeds cleanly, continue to post-update steps.
- If there are merge conflicts:
  - Run `git diff --name-only --diff-filter=U` to list conflicted files.
  - Show the list of conflicted files to the user.
  - Tell them:
    > "There are merge conflicts in the files listed above. This usually happens if you've modified files in `modules-core/` or other framework files."
    > "Please resolve the conflicts manually, then run `git add <file>` and `git commit` to complete the merge."
    > "Tip: Keeping your customizations in `modules-custom/` avoids most conflicts."
  - Do NOT run `git merge --abort` unless the user asks. Stop here.

### 8. Post-Update Tasks

After a successful merge:

1. **Check if dependencies changed**: Run `git diff HEAD~1..HEAD --name-only` and look for `package.json` or `package-lock.json`.
   - If changed, ask the user: "Dependencies have changed. Run `pnpm install` now?"
   - If they confirm, run `pnpm install`.

2. **Check for new SQL migrations**: Look for any new or changed `.sql` files in the diff.
   - If found, list them and tell the user:
     > "New database migration files were detected. Please review and run them manually in your SQL client of choice — Supabase Studio, pgweb, or `psql`."

3. **Check if module files changed**: Look for changes in `modules-core/` or `module.json` files.
   - If changed, run `pnpm generate-module-registry` automatically.

4. **Check for new environment variables**: Run `git diff HEAD~1..HEAD -- .env.example` to see if new env vars were added.
   - If changed, tell the user:
     > "`.env.example` has been updated with new environment variables. Please review and update your `.env` file."

### 9. Summary

Show a final summary:
> "ARI has been updated successfully!"
> Include: number of new commits merged, any action items remaining (pnpm install, SQL migrations, env vars).
> Remind the user to restart their dev server if it's running.

## Important Rules

- NEVER run SQL files automatically — always list them for the user to run manually.
- NEVER force-push, reset, or use any destructive git commands.
- NEVER modify `.env` files.
- NEVER touch `modules-custom/` — that directory belongs to the user.
- If anything goes wrong, explain clearly and let the user decide how to proceed.
